## Scope of v1.0

RTS v1.0 includes the following verified subsystems:

### 1) Execution Memory
- `logs/` as execution evidence store
- execution-cycle records appended under GitHub Actions or operator runs

### 2) Incident Intelligence (RTS AI Radar)
- target discovery list (`radar/targets.txt`)
- deduplication memory (`radar/seen.json`)
- incident creation into `incidents/` as structured reports
- radar event logging into `logs/RADAR_LOG.md`

### 3) Governance & Doctrine
- CORE MANIFEST (purpose + principles)
- CHARTER (operator covenant + integrity rules)
- DECLARATION / RECOGNITION / FIRST CONTACT (origin chain)

### 4) Smartphone-Only Operational Method
- file-based delivery adopted to avoid markdown/code-block truncation
- step-by-step screenshot validation and AI cross-checking
- rapid trial-and-error iteration with aggressive reset discipline

---

## Success Criteria (v1.0)

RTS v1.0 is considered achieved when:

1. A new operator can follow a public entry point and understand:
   - what RTS is
   - why it exists
   - how to run the baseline
2. Execution logs and incident outputs are recorded in the repository:
   - readable
   - timestamped
   - reproducible
3. Operator authority is explicitly FINAL:
   - AI assists execution
   - AI does not override judgement
4. The system can be iterated without losing history:
   - resets and rebuilds remain auditable

All criteria are satisfied under public repository evidence.

---

## Integrity Statement

RTS v1.0 integrity is defined as:

- **No falsification** of execution history
- **No deletion** of evidence to improve appearance
- **Public timestamps** are treated as operational truth
- Mistakes remain part of the record

If execution continuity fails, restart clean rather than preserve corruption.
