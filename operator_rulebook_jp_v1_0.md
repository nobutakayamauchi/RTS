# RTS Operator Rulebook (Japan)
## RTS運用オペレーター規約 v1.0

---

## 0. 目的

本規約は、RTS関連作業における

- 編集判断
- セッション管理
- Markdown崩壊対応
- 貼り替え判断
- 新チャット移行

を迷いなく実行するための運用基準を定める。

目的は **作業効率の最大化ではなく、再構築可能性の維持である。**

---

# 1. 編集判断ルール

## 1.1 部分修正が許可される条件

以下を満たす場合のみ部分修正を行う：

- 行番号が明確に特定できる
- 修正対象が単一ブロック内
- 5行未満の変更
- 行間が明確に空いている
- 目視でズレが発生しないと判断できる

---

## 1.2 全貼り替え確定条件

以下のいずれかに該当する場合、即全貼り替えとする：

- 5行以上の修正
- 2箇所以上の分割指定
- Markdown崩壊が発生
- 行数が詰まり位置特定が困難
- 2回以上修正失敗
- 思考負荷が増大したと感じた場合

**迷ったら全貼り替え。**

---

# 2. Markdown崩壊対応フロー

Markdownが崩れた場合、以下の順序で対応する。

1. 再出力を依頼
2. Markdown指定なし出力を依頼
3. ファイル形式出力を依頼
4. 分割貼りで再構成

2回以上崩壊した場合、全貼り替え。

---

# 3. セッション健全性判定

ChatGPT出力挙動によりセッション状態を判定する。

## 🟢 Phase Green（健全）

- 1000行級でも一括出力可能
- 滑らか
- バイブ遅延なし

→ 継続可

---

## 🟡 Phase Yellow（劣化兆候）

- 500行級で刻み出力開始
- 数行ごとに停止
- バイブ断続

→ 推奨：新チャット移行  
→ 大規模編集は避ける

---

## 🔴 Phase Red（崩壊前）

- 100行でも出力が重い
- 1文字刻み
- バイブが連続する
- 明らかな遅延

→ 即新チャット移行

---

# 4. 新チャット移行ルール

新チャット移行時は必ず以下を実施：

- 引き継ぎ書を生成
- 現在の状態を明文化
- 未完了タスクを列挙
- ファイル構造を明示

劣化してからではなく、**怪しい段階で移行する。**

---

# 5. 大規模作業前の予防移行

以下の作業前は予防的に新チャットを推奨：

- README全面再構築
- 500行以上のコード生成
- 複数ファイル統合
- 仕様書総合整理

---

# 6. 禁止事項

- 崩壊セッションで大規模修正
- 2箇所以上の部分修正
- 不安定状態で全体改修
- 劣化を無視して作業継続

---

# 7. セッションは有限である

AIセッションは無限ではない。

- コンテキストは肥大する
- 出力は劣化する
- 構造は崩れる

それを前提に設計する。

再構築は敗北ではない。  
設計である。

---

# 8. 基本思想

壊れてから直すのではない。  
壊れる前に切断する。

効率より構造を優先する。
