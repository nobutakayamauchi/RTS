# RTS Rulebook (Japan)
## RTS構造ルールブック v1.0

---

## 0. 本書の目的

本書はRTSの設計思想、構造原則、保証範囲、および運用背景を明文化するものである。

RTSは単なるログシステムではない。
RTSは再構築可能性を最優先とする構造基盤である。

本書はその統治原則を定める。

---

# 1. Structural Principles（構造原則）

RTSは以下の原則に基づく。

### 1.1 Reconstructability First
出力の再現ではなく、
**意思決定状態の再構築可能性**を優先する。

### 1.2 Append-Only Ledger
セッション記録は追記型とする。
既存履歴は改変しない。

### 1.3 Deterministic Aggregation
月次インデックスは決定論的に生成される。

### 1.4 Breakpoint-Based Sealing
変異密度が閾値を超えた場合、
構造は封印され、ESCが生成される。

---

# 2. Artifact Contract（成果物契約）

RTSは以下の成果物を生成する。

- Session Ledger（JSONL）
- Monthly Index
- Snapshot
- Evidence Snapshot（ESC）

検証は成果物からのみ行う。

成果物から検証できないものはRTSではない。

---

# 3. Mutation Handling（変異処理）

RTSはノイズに反応しない。
構造変異に反応する。

Breakpoint検出時：

1. 状態を固定
2. ESC生成
3. Snapshot保存
4. 不変参照化

監視ではない。
構造封印である。

---

# 4. Non-Goals（保証しないもの）

RTSは以下を保証しない：

- 判断の倫理的正しさ
- 法的正当性
- 意味的妥当性
- 結果の成功
- セマンティック完全性

RTSが保証するのは：

構造的再構築可能性のみである。

---

# 5. Why These Rules Exist（本ルールの背景）

RTSは制約下で開発されている。

- 単一デバイス開発
- GitHubネイティブ編集
- ローカルIDE非依存
- ブラウザベース運用
- AI協働執筆

これらは偶然ではない。

これらの制約が
RTSの運用原則を形成している。

なぜ全貼り替えを優先するのか。
なぜセッションを早期に切断するのか。
なぜ部分修正を制限するのか。

それは、

- 制約環境下での多行パッチは不安定であるため
- コンテキスト蓄積がAI出力の信頼性を劣化させるため
- 構造明瞭性が局所最適より重要であるため

である。

より効率的に見える手法は存在する。

しかしRTSは、

局所効率より構造的整合性を優先する。

---

# 6. Constraint as Discipline（制約は設計境界である）

制約は欠陥ではない。
設計境界である。

RTSは制約下でも再構築可能でなければならない。

- ローカル環境がなくても理解できる
- IDEがなくても追跡できる
- 外部依存がなくても再現できる

これがRTSの最低条件である。

---

# 7. セッションも有限である

AI協働セッションは無限ではない。

- コンテキストは肥大する
- 出力は劣化する
- レイテンシは増大する

従って、

再構築は敗北ではない。
設計である。

---

# 8. 最終原則

RTSは完璧性を目指さない。

RTSは検証可能性を目指す。

再構築できないものは、守れない。
守れないものは、拡張できない。

以上。
